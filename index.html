O modelo de avaliação

O Base2 Score v3 avalia maturidade em qualidade de software em 6 temas com 37 perguntas. O resultado principal é o radar de maturidade por tema — cada tema recebe um nível independente (Inicial → Avançado) sem agregação forçada entre dimensões. O score numérico (0–100) existe como referência secundária para acompanhamento histórico.

Os pesos de cada tema são derivados automaticamente das respostas sobre o contexto real do sistema: frequência de mudanças, criticidade operacional, complexidade de integração e dependência de jornadas de usuário — em vez de um perfil fixo predefinido.

Os 6 temas

Testes de Unidade — proteção no nível de código, feedback rápido, base para evolução segura.

Testes de Integração — cobertura das interações entre componentes, serviços, filas e banco.

Testes de Sistema — regressão funcional, estabilidade de ambiente e pós-deploy.

End-to-End — validação estratégica de jornadas críticas sem flakiness.

Qualidade de Engenharia — quality gates, code review, métricas integradas ao processo.

Qualidade Operacional — confiabilidade em produção, incidentes, segurança, observabilidade.

Níveis de maturidade

0–39%Inicial — práticas inexistentes ou muito pontuais. Alto risco operacional.
40–59%Em desenvolvimento — algumas práticas existem, mas sem consistência.
60–79%Intermediário — práticas consolidadas, automação relevante, gaps pontuais.
80–100%Avançado — qualidade integrada ao processo de engenharia.

Riscos críticos

Certas perguntas são marcadas como gatilhos de risco. Se respondidas com "Não", geram uma seção de Riscos Críticos no relatório — independente do score total.

Esses gatilhos representam ausências que expõem o sistema a falhas graves mesmo que os demais indicadores sejam bons. Exemplos: ausência de CI bloqueante, ausência de regressão definida, ausência de processo de incidentes, ausência de observabilidade em produção.

Pesos derivados de risco

Em vez de escolher um perfil fixo, você responde 4 perguntas sobre o sistema. As respostas ajustam os pesos proporcionalmente: frequência de mudanças aumenta o peso de Unidade e Integração; criticidade operacional aumenta Operacional e Sistema; complexidade de integrações aumenta Integração; dependência de jornadas aumenta E2E.

O perfil de referência "mais próximo" é exibido para contexto, mas não determina os pesos.

Boas práticas de uso

• N/A deve ser usado apenas quando o tema genuinamente não se aplica ao contexto.

• "Parcial" = prática existente mas incompleta ou inconsistente.

• O radar por tema é o resultado principal — o número é referência secundária.

• Reavaliações trimestrais ou semestrais permitem acompanhar a evolução.

• Use o campo de evidência em cada pergunta para registrar o que foi observado.
